Rolling Restart Helper

Step-by-step guided workflow for safely restarting cluster nodes one at a time.

What is this and when do I need it?

A rolling restart lets you restart every node in an Elasticsearch cluster one at a time, without downtime or data loss. You need it when:

  • Applying OS or JVM patches that require a process restart
  • Upgrading Elasticsearch to a new minor/patch version
  • Changing JVM settings (heap size, GC options) that require a restart
  • Replacing hardware or migrating nodes to new machines

Why not just restart all nodes at once? Because Elasticsearch would try to rebalance shards for every node that goes down, causing massive unnecessary data movement, high disk I/O, and potential data unavailability.

This wizard automates the official rolling restart procedure:

  1. Check health — make sure the cluster is green (or at least yellow) before starting
  2. Disable allocation — tell ES not to move shards while a node is down
  3. Flush — write in-memory data to disk so recovery is faster
  4. Restart the node — you do this outside Cerebro (systemctl, docker, etc.)
  5. Re-enable allocation — let ES assign shards back to the restarted node
  6. Wait for green — confirm all shards are recovered before moving to the next node

Repeat steps 2–6 for each node in the cluster.
